# Execution Plan: Doc Integrity + Doc Gardening

Updated: 2026-02-23

## Objective
Keep `docs/` accurate, navigable, and agent-usable by enforcing structure and freshness with automated checks and recurring maintenance.

## Non-goals
- Writing all missing documentation upfront.
- Enforcing a specific doc toolchain across all future repos (this is a template; downstream repos will adapt).

## Scope
- Add dedicated linters and CI jobs to validate the knowledge base is:
  - Up to date (or explicitly marked as stale with a remediation path)
  - Cross-linked (no broken internal links; indexes stay current)
  - Structured correctly (required files/sections present; consistent layout)
- Add a recurring “doc-gardening” agent that scans for stale/obsolete docs and opens fix-up PRs.

## Proposed invariants (v1)
- `AGENTS.md` links to existing docs (no dangling references).
- `docs/design-docs/index.md` links to all design docs in `docs/design-docs/` (except `index.md` itself).
- `docs/product-specs/index.md` links to all specs in `docs/product-specs/` (except `index.md` itself).
- No broken relative links inside `docs/` (markdown links to repo files resolve).
- Generated docs in `docs/generated/` are either:
  - Regenerated in CI, or
  - Clearly labeled with how/when they are generated.
- Docs that describe behavior with high drift risk include a lightweight freshness marker:
  - Example: `Last reviewed: YYYY-MM-DD` (policy for enforcement TBD).

## Milestones
1. Define doc structure rules (what is required vs. recommended).
2. Implement a doc-lint script (fast, deterministic) that checks the invariants.
3. Wire doc-lint into CI as a required check.
4. Define the doc-gardening process:
   - What “stale” means (signals and thresholds)
   - How findings become PRs (labels, PR template, review expectations)
5. Implement doc-gardening as a recurring agent run (or scheduled CI) that opens fix-up PRs.

## Implementation notes
- Prefer simple, repo-local tooling (a single script) over heavy frameworks.
- Keep “policy” in docs; keep “enforcement” in code/CI.
- Start with structural/link checks first; add semantic drift checks incrementally.

## Validation
- Doc-lint runs in CI and fails on:
  - Broken links
  - Missing required index entries
  - Dangling references from `AGENTS.md`
- A sample “stale doc” is detected and a fix-up PR is generated by the doc-gardening run (dry-run acceptable for v1).

## Rollout
- Land doc-lint + CI gating first (low risk).
- Add doc-gardening in non-blocking mode initially (opens PRs but does not block merges).

## Risks
- Over-enforcement creates busywork: keep v1 rules minimal and high-signal.
- False positives for links/paths: ensure lint rules match how docs are authored in this template.
